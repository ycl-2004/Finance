# QM Financial Learning Hub 产品与实施规划

> 日期：2026-04-30
> 状态：规划草案，待用户确认后进入代码实现
> 当前目标：先做无 AI 的公开中文金融学习网站，后续再升级为 AI 学习教练和知识库问答系统

## 1. 项目一句话

QM Financial Learning Hub 是一个面向中文小白和未来客户教育场景的加拿大金融知识中心。它把现有 Markdown 知识库转成可公开访问、可持续维护、可系统学习、可展示专业度的网页应用。

## 2. 已确认决策

| 决策 | 结论 |
| --- | --- |
| 产品名称 | QM Financial Learning Hub |
| 第一版是否公开 | 完全公开访问 |
| 内容语言 | 中文为主，保留必要英文金融术语 |
| 第一版是否接 AI | 不接 AI，先做学习网站和知识库体验 |
| 产品气质 | 专业顾问机构 / 客户教育中心 |
| 第一版用户 | 自己系统学习为主，同时可给中文客户阅读 |
| 长期方向 | 完整知识库体系，可持续扩展，可升级 AI 问答、测验、学习教练 |

## 3. 核心问题

我们要解决的问题不是“如何做一个漂亮网页”，而是：

> 如何把一套顾问业务背后的加拿大金融知识，变成一个中文小白可以按路径学习、客户可以安全阅读、未来可以持续更新和扩展的专业知识中心？

## 3.1 关键产品约束：用户路径优先

这个项目最大的产品风险是自然滑向“高级文档站”。如果执行顺序是 Markdown -> 渲染 -> 页面，结果大概率会变成“文件 -> 页面”，而不是“用户路径 -> 页面”。

因此第一版必须强制采用用户路径优先：

1. 先设计首页 dashboard。
2. 再设计学习路线。
3. 再设计生活场景 / 按问题学习入口。
4. 最后才把 Markdown 文章页作为承载层接入。

文章页很重要，但它不是产品骨架。真正决定这个产品不像普通文档站的，是首页、学习路线、生活场景、案例练习如何引导用户理解和行动。

## 3.2 2026-04-30 定位升级：会前准备平台

最新产品定位从“金融知识库 / 学习中心”升级为“客户在见金融顾问前的准备平台”。

这次调整的核心不是减少内容，而是改变入口顺序：

1. 首页先说明能帮用户做什么：准备关键问题与资料。
2. 用户优先进入“开始规划”，按新移民、买房、家庭、房贷续约、退休、企业主等场景选择。
3. 每个场景必须给出行动流程：现在要做什么、常见错误、需要准备什么、应该问顾问什么。
4. “资料准备”成为核心产品：可勾选 checklist、本机保存进度、支持打印或保存为 PDF。
5. “学习中心”降级为辅助模块：当用户在资料准备和场景流程中遇到不懂概念时，再回到文章补知识。
6. “关于我们”必须明确平台身份、持牌边界和不提供的服务。

验收标准：

- 用户 3 秒内知道平台是会前准备工具。
- 用户 10 秒内能点击一个生活场景。
- 用户 1 分钟内能进入资料清单并开始勾选。

## 4. 用户画像

### 4.1 主要用户：你自己

目标：

- 系统学习加拿大金融顾问相关知识。
- 从“知道概念”进阶到“能解释给别人听”。
- 能持续补充、修改、复盘知识库。
- 能把成品作为作品案例展示。

成功体验：

- 打开网站后知道今天该学什么。
- 每个主题知道先读哪篇、后读哪篇。
- 遇到英文术语可以快速查中文解释。
- 学完一个主题后可以用案例检验自己是否真的理解。

### 4.2 次要用户：中文客户 / 潜在客户

目标：

- 用中文理解加拿大金融规划常见主题。
- 在见顾问前先知道基本概念和资料准备方式。
- 明白哪些内容是教育信息，哪些需要专业人士确认。

成功体验：

- 能快速找到自己的问题入口，例如买房、注册账户、保险、退休。
- 读完后知道要准备什么资料、问顾问什么问题。
- 不会被误导成“这是个人建议”。

### 4.3 未来用户：合作伙伴 / 面试官 / 展示对象

目标：

- 看出这个项目不是普通文档，而是完整产品。
- 看出你有内容架构、金融边界、用户教育、前端产品设计和长期维护能力。

成功体验：

- 3 分钟内理解产品定位。
- 5 分钟内看出内容体系和信息架构。
- 10 分钟内看到案例、术语、资料清单和学习路径都能串起来。

## 5. 产品原则

1. 教育优先，不做个人建议。
2. 内容可信，来源清楚，更新时间清楚。
3. 小白友好，但不幼稚化。
4. 顾问机构气质，专业、克制、清晰。
5. 先做可用的知识中心，再做 AI。
6. Markdown 继续作为内容源，降低长期维护成本。
7. 每个页面都要回答“用户下一步该看什么”。
8. 会变化的信息必须标记 review date 和核对提醒。

## 6. 不做事项

第一版明确不做：

- 不做 AI 问答。
- 不做登录系统。
- 不保存真实客户资料。
- 不做客户风险问卷提交。
- 不推荐具体证券、基金、保险、贷款或税务结构。
- 不做真实税务、房贷、退休计算器。
- 不做后台 CMS。
- 不做英文完整站点。
- 不把官网做成营销落地页。

原因：

- 当前最重要的是建立知识体系和公开学习体验。
- 金融建议、客户资料和计算器会带来合规、准确性和隐私风险。
- AI 在没有内容结构、引用、边界之前接入，容易变成不可靠的聊天框。

## 6.1 第一版投入占比

为了避免做成普通文档站，第一版的精力分配不应该按“文章数量”来算，而应该按“用户是否能被引导学习”来算。

建议占比：

| 模块 | 占比 | 为什么 |
| --- | --- | --- |
| 首页 dashboard、学习路线、生活场景 | 35% | 这是产品骨架，决定用户是不是能从问题进入学习 |
| 内容模型、metadata、校验 | 20% | 保证知识库长期可维护，不靠人工拼页面 |
| 文章页和主题知识库 | 20% | 承载深度内容，但不应该主导产品形态 |
| 术语表、搜索、资料清单 | 10% | 作为学习工具，帮助用户快速定位 |
| 案例练习 | 10% | 把阅读变成理解和应用，提升展示价值 |
| 边界、QA、部署 | 5% | 保证公开访问时可信、安全、稳定 |

验收原则：

- 如果首页、学习路线、生活场景没有完成，即使所有 Markdown 都能渲染，也不能算 MVP 完成。
- 如果用户只能通过主题或文件列表找内容，说明产品仍然偏文档站。
- 如果客户可以从“我遇到的问题”进入内容，才算客户教育中心成立。

## 7. 现有资产

项目已有 40 个 Markdown 文件，覆盖：

- 公司基础资料
- 金融术语表
- 客户资料取得流程
- 综合财务规划
- 投资规划和投资组合策略
- 加拿大注册账户
- 保险和风险管理
- 房贷和贷款方案
- 税务优化和资产保护
- 退休规划与财富传承
- 小企业 / 企业主财务咨询

当前内容优点：

- 主题完整。
- 中文小白友好。
- 已经有内容边界。
- 多数文件已有 `topic`、`audience`、`last_reviewed` frontmatter。
- 来源放在底部，适合学习阅读。

当前需要补强：

- 每篇文章缺统一难度、预计阅读时间、学习目标、关键词、相关主题。
- 缺学习路径。
- 缺生活场景 / 客户教育入口。
- 缺案例练习。
- 缺测验和复习机制。
- 缺网页产品层的信息架构。

## 8. 推荐技术路线

### 8.1 推荐方案：Next.js + Markdown 内容源

选择：

- 使用 Next.js / React 做网页应用。
- 保留 `knowledge/` 作为核心内容源。
- 通过脚本读取 Markdown frontmatter 和正文，生成网站页面。
- 第一版可以静态生成，公开部署到 Vercel 或类似平台。
- 搜索先用静态全文搜索，例如 Pagefind 或构建时生成索引。

适合原因：

- 比纯文档站更适合做学习仪表盘、案例、筛选、未来 AI 入口。
- 比低代码工具更像完整作品。
- Markdown 内容不被锁进数据库，长期可维护。
- 后续可以逐步加入登录、AI、进度、测验、向量搜索。

### 8.2 备选方案：Docusaurus

适合：

- 想最快把 Markdown 变成公开文档站。
- 不追求强学习路径和产品化体验。

不推荐作为最终第一版的原因：

- 更像文档网站，不像金融顾问知识中心产品。
- 自定义学习仪表盘、案例练习、测验体验会更受限制。

### 8.3 现成工具定位

| 工具 | 适合什么 | 是否作为最终产品 |
| --- | --- | --- |
| Logseq | 你自己的双向笔记和学习复盘 | 否，只作个人学习辅助 |
| Anki | 术语、账户规则、易错点记忆卡片 | 否，可后续导出卡片 |
| AnythingLLM | 快速验证文档问答效果 | 否，可做 AI 原型 |
| Khoj | 个人 second brain 和文档问答 | 否，可做个人学习辅助 |
| Dify | 快速搭 AI 工作流和 RAG | 后续可做 AI 原型，不做第一版核心 |
| Docusaurus | 快速公开文档站 | 可做备选，不作为推荐主线 |

## 9. 视觉方向

### 9.1 设计参考

主参考：Mintlify 风格
原因：适合文档型、知识型、阅读型产品，左侧导航、正文层级、搜索入口和内容组织成熟。

辅参考：Stripe 风格
原因：金融产品需要信任、精致、专业和克制，Stripe 的层级、留白、可信赖感可以作为参考。

### 9.2 设计关键词

- 专业
- 清晰
- 稳重
- 温和
- 高信任感
- 信息密度适中
- 中文阅读友好

### 9.3 颜色建议

不要做成大面积紫色、深蓝、金色、渐变金融模板。

建议：

- 背景：暖白或浅灰白。
- 主文字：接近黑色的中性灰。
- 辅助文字：中灰。
- 主色：沉稳蓝绿或深青色，体现专业和可信赖。
- 强调色：少量用于标签、进度、重点提示。
- 警示色：用于“需核对”“非建议”“易过期信息”。

### 9.4 页面布局原则

- 首页不是营销 hero，而是知识中心 dashboard。
- 桌面端优先三栏信息结构：导航、内容、辅助信息。
- 移动端优先阅读体验：顶部搜索、折叠导航、正文优先。
- 卡片只用于主题入口、案例、测验，不把所有 section 都塞进卡片。
- 每篇文章顶部必须显示：主题、难度、预计阅读时间、最后校准时间、内容边界。
- 在写正式页面代码前，必须先完成 3 个低保真 wireframe：首页 dashboard、学习路线、文章页。推荐同时补一个生活场景详情页 wireframe。
- Wireframe 验收通过前，不进入文章渲染实现。否则 UI 容易退化成“把 Markdown 放进容器里”。

## 10. 信息架构

第一版信息架构要按用户路径组织，而不是按文件夹组织。

推荐主结构：

1. 首页
2. 开始学习
3. 生活场景
4. 主题知识库
5. 工具
6. 案例练习
7. 关于与使用边界

其中：

- 开始学习：核心入口，包含学习路线、阶段目标、推荐下一步。
- 生活场景：原“客户教育”，改成更自然的用户语言，也可以叫“按问题学习”。
- 主题知识库：深度内容入口，适合已经知道自己要查哪个主题的人。
- 工具：术语表、资料清单、未来可能的测验和 Anki 导出。
- 案例练习：进阶训练入口。

实施优先级：

1. 首页 dashboard
2. 学习路线
3. 生活场景 / 按问题学习
4. 文章页
5. 主题知识库
6. 术语表、搜索、案例、资料清单

### 10.1 首页

目的：

- 让用户一眼知道这是中文加拿大金融学习中心。
- 让小白知道从哪里开始。
- 让客户知道可以按场景阅读。
- 让展示对象看到体系完整。

页面内容：

- 产品名称和一句话定位。
- “从这里开始”学习入口。
- 8 大主题入口。
- 常见生活场景 / 按问题学习入口。
- 最新校准时间提示。
- 教育用途声明。

验收标准：

- 新用户 10 秒内知道网站是做什么的。
- 首页首屏出现产品名、学习入口、搜索或主题入口。
- 不出现具体产品推荐。
- 移动端首屏不拥挤，主 CTA 明确。
- 首页必须优先引导“从哪里开始学”和“按问题进入”，不能只是主题卡片集合。

### 10.2 学习路线

目的：

- 把 40 个知识文件变成有顺序的学习系统。

建议路线：

1. 金融地图和术语基础
2. 综合财务规划
3. 客户资料和规划流程
4. 注册账户
5. 投资规划
6. 保险和风险管理
7. 房贷和买房
8. 税务优化和资产保护
9. 退休和财富传承
10. 小企业 / 企业主财务咨询

每个学习阶段展示：

- 学习目标
- 适合谁
- 前置知识
- 必读文章
- 选读文章
- 学完后应该能回答的问题
- 对应案例练习

验收标准：

- 完全小白可以按路线连续学习。
- 每个阶段至少有 1 个学习目标和 2 个检查问题。
- 所有现有主题都能映射到某个阶段。
- 用户不会被直接丢进 40 个文件列表里。

### 10.3 主题知识库

目的：

- 保留顾问业务视角，让内容按服务方向检索。

主题：

- 综合财务规划
- 投资规划和投资组合策略
- 加拿大注册账户
- 保险和风险管理
- 房贷和贷款方案
- 税务优化和资产保护
- 退休规划与财富传承
- 小企业 / 企业主财务咨询

每个主题页展示：

- 主题定义
- 解决什么问题
- 适合谁读
- 文章列表
- 相关术语
- 相关生活场景
- 需要专业确认的问题

验收标准：

- 每个主题有独立 landing page。
- 每个主题至少列出 2 篇内容，现有不足则标记待补。
- 主题页不只是文件列表，要有学习引导。
- 点击文章后能回到主题页或下一篇。

### 10.4 文章页

目的：

- 提供正式、可阅读、可引用的学习内容。

文章页结构：

- 面包屑导航。
- 标题。
- 主题标签。
- 难度。
- 预计阅读时间。
- 最后校准时间。
- 内容边界提示。
- 正文。
- 小白重点。
- 常见误区。
- 相关术语。
- 相关案例。
- 下一篇推荐。
- 资料来源。

验收标准：

- 所有 Markdown 都能正常渲染。
- 表格在移动端可横向滚动或重排，不溢出页面。
- 来源链接明显但不打断正文阅读。
- 每篇文章有最后校准时间。
- 会变化的信息有“实际使用前需核对”提示。

### 10.5 术语表

目的：

- 让中文用户快速理解英文金融术语。

功能：

- 按分类查看。
- 搜索英文或中文。
- 每个术语显示英文、中文、解释、相关主题。
- 支持从文章内点击术语跳转。

验收标准：

- 能搜索 TFSA、RRSP、KYC、MER、mortgage 等词。
- 搜索中文“免税储蓄账户”也能找到 TFSA。
- 每个术语至少属于一个分类。
- 术语解释不写成建议。

### 10.6 生活场景 / 按问题学习

目的：

- 把知识库从“按学科组织”转成“按真实问题组织”。
- 这是客户教育中心的用户可见入口，但导航命名不使用“客户教育”，避免听起来像内部业务分类。

第一版生活场景：

1. 我刚来加拿大，应该先了解什么金融基础？
2. 我想买第一套房，应该先准备什么？
3. TFSA、RRSP、FHSA 有什么不同？
4. 我有孩子，RESP 要怎么理解？
5. 我有房贷快续约，要注意什么？
6. 我想了解保险，不知道从哪里开始。
7. 我快退休了，要先看哪些资料？
8. 我是小企业主，个人和公司财务要分清什么？

每个场景页面展示：

- 谁适合看。
- 先理解的 3-5 个概念。
- 需要准备的资料。
- 可以问顾问的问题。
- 相关知识库文章。
- 不应该直接下结论的地方。

验收标准：

- 客户不需要知道专业分类，也能从问题进入内容。
- 每个场景都有资料准备和顾问问题清单。
- 场景页明确教育边界。
- 不出现“你应该买 / 投 / 借 / 选”的指令。
- 导航和页面标题优先使用“生活场景”或“按问题学习”，内部文档中可以继续称为 client education scenario。

### 10.7 案例练习

目的：

- 帮你从“读懂”变成“能解释和判断问题边界”。

第一版案例：

1. 新移民家庭建立财务规划地图。
2. 首次买房者比较 FHSA、RRSP HBP、现金流和房贷资料。
3. 年轻家庭评估保险需求和风险缺口。
4. 中年高收入家庭比较 TFSA、RRSP、非注册账户学习顺序。
5. 临近退休客户整理 CPP、OAS、RRSP/RRIF 和现金流问题。
6. 小企业主区分个人现金流、公司税务、保险和退出风险。

每个案例结构：

- 背景。
- 已知资料。
- 缺失资料。
- 学习任务。
- 应该先问的问题。
- 可能涉及的知识点。
- 不可直接给建议的边界。
- 参考答案或思路。

验收标准：

- 每个案例都能链接到至少 3 篇知识文章。
- 每个案例都有“缺失资料”部分。
- 参考答案是分析框架，不是具体产品建议。
- 案例能帮助用户检查自己是否理解多个主题之间的关系。

### 10.8 资料清单

目的：

- 让客户知道金融规划前资料从哪里拿。

内容来源：

- `knowledge/client-document-source-map.md`

页面结构：

- 税务资料
- CRA / My Account
- My Service Canada Account
- 银行现金流
- 投资和注册账户
- 信用和债务
- 房贷和房产
- 保险
- 退休
- 遗产和失能
- 企业主资料

验收标准：

- 客户能按类别找到“去哪里拿、拿什么、注意什么”。
- 明确提醒不要共享密码、完整 SIN、完整账号。
- 资料都强调日期和时效性。
- 页面适合会议前发给客户预习。

### 10.9 关于与使用边界

目的：

- 建立信任，同时降低误解风险。

必须包含：

- 本站是教育用途。
- 不构成投资、保险、房贷、税务或法律建议。
- 不推荐具体产品。
- 会变化的信息需重新核对。
- 来源说明。
- 内容更新时间。

验收标准：

- 首页、文章页、生活场景页都能访问边界声明。
- 边界声明不是藏在页脚小字。
- 用户不会误以为网站在替代持牌顾问、会计师、律师或贷款专业人士。

## 13. 实施阶段

### Phase 0：规划确认

目标：

产出：

- 用户确认公开访问、中文为主、专业顾问机构风格。
- 用户确认先做完整知识中心，再做 AI。

### Phase 1：用户路径和 Wireframe

目标：

- 先确定用户如何进入学习，防止项目变成“Markdown 文件展示器”。

工作内容：

- 画首页 dashboard wireframe。
- 画学习路线 wireframe。
- 画文章页 wireframe。
- 推荐补充生活场景详情页 wireframe。
- 确认首页首屏的信息优先级。
- 确认学习路线如何把小白带到第一篇文章。
- 确认生活场景如何从真实问题链接到知识库。
- 确认文章页如何支持学习，而不是只展示正文。

验收标准：

- 首页 wireframe 不是营销 landing page，也不是主题卡片堆叠。
- 学习路线 wireframe 能显示阶段、目标、必读、检查问题。
- 生活场景 wireframe 能显示“适合谁、先理解什么、准备什么资料、问顾问什么”。

- 每个主题有排序和描述。
- 至少 8 个生活场景完成。
- 至少 6 个案例练习完成。
- 每个学习阶段都能指向具体文章。
- 每个生活场景都能指向具体文章、术语和资料清单。

### Phase 9：后续 AI 升级

目标：

- 在内容体系稳定后，把 AI 变成学习教练，而不是普通聊天框。

- 自动生成测验题。

- 只能基于知识库和来源回答。
- 必须显示引用。
- 不能输出具体投资、保险、贷款、税务或法律建议。

验收标准：

- AI 回答能引用具体文章。
- AI 不编造来源。
- AI 遇到个人化建议问题时会拒绝直接建议并引导收集资料。
- AI 输出可被日志或测试样例检查。

## 16. 风险和应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 内容太多导致第一版拖慢 | 上线慢 | 第一版只做现有 40 篇，不无限扩写 |
| 变成普通文档站 | 展示价值不足，用户只能按文件找内容 | 强制先做首页、学习路线、生活场景，再做文章页 |
| 内容和 UI 脱节 | UI 只是展示 Markdown，不能引导学习 | Phase 1 先画首页、学习路线、文章页 wireframe |
| 过早接 AI | 不稳定、不可信 | AI 放到 Phase 9 |
| 金融内容被误解成建议 | 合规风险 | 每页边界、用语检查、场景只给框架 |
| Markdown metadata 不统一 | 页面生成不稳定 | Phase 2 做内容校验 |
| 中文长文本移动端难读 | 用户体验差 | 移动端优先检查文章页和表格 |
| 搜索效果弱 | 找内容困难 | 先做关键词搜索，后续再做语义搜索 |

## 17. 后续可扩展方向

### 17.1 测验系统

- 每篇文章 3-5 道题。
- 答错显示对应知识段落。
- 不需要登录时可用 localStorage 保存进度。

### 17.2 Anki 导出

- 把术语表和易错点导出为 Anki 卡片。
- 适合你自己背诵和复习。

### 17.3 AI 学习教练

- 根据文章生成解释。
- 根据用户回答判断理解。
- 只引用知识库。

### 17.4 顾问会议辅助

- 按生活场景生成资料准备清单。
- 生成“可以问顾问的问题”。
- 不保存客户隐私。

### 17.5 多语言

- 第二阶段可加入英文摘要。
- 不建议第一版双语全量翻译，工作量过大。

Phase 1 通过后，再进入内容标准化和网页应用开发。这样后面的前端会围绕用户路径展开，而不是把文件直接变成页面。
